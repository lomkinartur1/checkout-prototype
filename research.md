# Чекаут интернет-магазина электроники — исследование и решения

Роль: PM чекаута. Задача: спроектировать оптимальный путь оформления заказа
(доставка → данные покупателя → оплата) для B2C интернет-магазина.
Результат — кликабельный прототип и обоснование решений.

- **Прототип:** https://lomkinartur1.github.io/checkout-prototype
- **Репозиторий:** https://github.com/lomkinartur1/checkout-prototype

---

## 1. Контекст и вводные

| Параметр | Решение |
|---|---|
| Категория | Электроника и бытовая техника (высокий средний чек) |
| География | Россия |
| Платформа | Mobile-first web, адаптируется до десктопа |
| Тип магазина | Новый интернет-магазин — своей базы лояльных нет |
| Основной пользователь | Новый / неавторизованный покупатель |
| Авторизация | Обязательна перед чекаутом (Яндекс ID или телефон+OTP), гость — fallback |
| Доставка | Курьер + партнёрские пункты выдачи |
| Оплата | Карта, СБП (−3%), рассрочка (Сплит) |

**Главное продуктовое допущение.** Магазин новый, поэтому большинство трафика —
холодные пользователи с пустым профилем. При этом **Яндекс ID подтягивает данные
не из нашего магазина, а из экосистемы** (адрес из профиля, карта из Яндекс Пэй).
Поэтому даже новый для нас пользователь через Яндекс ID получает автозаполнение,
а остальные вводят данные вручную. Решение осознанное: авторизация через экосистему
даёт пользователю меньше ввода, а магазину — чистый чекаут без гостевых edge-кейсов.

## 2. Методология

1. **Анализ конкурентов** — чекауты российских и зарубежных сервисов: Ozon, Яндекс
   Маркет, Wildberries, Lamoda, Avito Доставка, Эльдорадо, Самокат, Яндекс Лавка,
   Яндекс Путешествия, Aviasales, Amazon, такси. Экраны разобраны по юзкейсам
   (адрес/доставка, получатель, оплата, доверие, сервисные опции, edge-кейсы);
   результат разбора — в разделах 3–4 ниже.
2. **Причины отказа от чекаута** — исследование Baymard Institute (мировая база по
   брошенным корзинам).
3. **Синтез** — отбор паттернов, которые решают наши задачи, и отсев антипаттернов.
4. **Прототипирование и итерации** — кликабельный HTML-прототип, доработка по замечаниям.

## 3. Что взяли у конкурентов и от чего отказались

**Взяли (паттерны, доказавшие себя у лидеров):**
- Sticky-кнопка оплаты с суммой всегда на виду — Яндекс Такси, Самокат
- Urgency-строка с дедлайном «Закажите сегодня до 14:00 — привезём завтра» — Яндекс Маркет
- Горизонтальный скролл дат доставки — Яндекс Маркет, Lamoda
- Скидка за СБП прямо в карточке метода оплаты — Lamoda, Эльдорадо
- Рассрочка с графиком платежей без переплат — Маркет, WB, Ozon, Avito
- Выбор пункта выдачи на карте + поиск — WB, Ozon, Lamoda
- Опции курьера (оставить у двери, не звонить, комментарий) — Самокат, Лавка, Маркет
- Блок доверия к оплате (безопасность, возврат, гарантия) — Avito Доставка
- Быстрый вход одним тапом через профиль — WB, Яндекс Такси

**Сознательно избежали (антипаттерны):**
- Отдельный пустой экран выбора доставки без цен и сроков — Lamoda
- Full-screen upsell внутри чекаута — Amazon Prime
- Включённый по умолчанию платный opt-in — Ozon ProZaPass
- Дублирование выбора оплаты в разных местах экрана — Ozon
- Снекбар поверх важного контента — Ozon

## 4. Причины отказа от чекаута (Baymard) и как мы их закрыли

| Причина брошенной корзины | Решение в прототипе |
|---|---|
| Скрытые дополнительные расходы | Доставка и итог видны заранее; в «Итого» построчная разбивка |
| Обязательная регистрация | Гостевой checkout как fallback; аккаунт создаётся после заказа, не до |
| Непонятная доставка | Даты, сроки и цена показаны сразу; ПВЗ со сроком хранения |
| Длинный / сложный процесс | Один экран чекаута, автоподстановка данных, smart-кнопка ведёт по шагам |
| Недоверие к оплате | Блок «Безопасная оплата · Возврат 14 дней · Гарантия» рядом с кнопкой |

## 5. Ключевые продуктовые решения

1. **Авторизация до чекаута.** Убирает граничные случаи гостевого заказа: данные
   подтягиваются, чекаут чистый. Гость доступен как запасной путь, чтобы не терять
   тех, кто не хочет входить.

2. **Один экран чекаута** (Доставка → Товары → Оплата → Итого) вместо пошагового
   визарда. Меньше переходов — меньше точек отвала. Структура линейна и предсказуема.

3. **Вход определяет состояние.** Яндекс ID → всё заполнено; телефон/гость → пустые
   поля и ввод с нуля. Прототип демонстрирует оба пути — это и есть доказательство
   ценности авторизации.

4. **Smart CTA вместо мёртвой блокировки.** Если данных не хватает, кнопка не серая
   и неактивная, а ведёт по шагам: «Указать адрес» → «Добавить получателя» →
   «Оплатить 67 480 ₽». Пользователь всегда видит следующий шаг.

5. **Курьер по умолчанию.** Для дорогой электроники доставка до двери приоритетнее
   ПВЗ; пункт выдачи доступен переключателем.

6. **Адрес и пункт выдачи — вручную или на карте.** Части пользователей проще найти
   точку на карте; даём оба способа. Для ПВЗ — поиск и карта с пинами.
   *(В прототипе карта — демонстрация паттерна; реальная интеграция карт вне scope теста.)*

7. **Получатель: телефон обязателен, имя — нет.** При входе по номеру телефон уже
   известен — не просим вводить его повторно. Для доставки достаточно телефона,
   имя опционально: на одно обязательное поле меньше.

8. **Оплата вынесена в отдельный денежный виджет** (способ оплаты → итого → кнопка).
   На десктопе он липкий и виден без скролла. Промо рассрочки и Сплита — там, где
   пользователь принимает решение об оплате, а не в подвале экрана.

9. **СБП со скидкой −3% бейджем в карточке метода** — выгода видна в момент выбора.
   Рассрочка при выборе раскрывает график: 6 платежей по 11 247 ₽, без переплат.

10. **Urgency на доставке** — дедлайн «закажите до 14:00» подталкивает завершить заказ.

## 6. Scope: что осознанно НЕ делали

Что осознанно не делали и почему:

| Не делаем | Почему |
|---|---|
| Промокоды в прототипе | Поддерживаются продуктом, но не нужны для демонстрации флоу |
| Бонусы / программа лояльности | Магазин новый, базы лояльных нет |
| B2B / оформление на юрлицо | Другой сценарий и слот |
| Подписки, кошелёк, чаевые, подарочная упаковка | Не профиль нового магазина электроники |
| Агрессивный upsell | Ломает намерение купить |
| Характеристики товара в чекауте | Решаются на карточке товара |

## 7. Edge cases

- **Ошибка оплаты** — отдельный экран «Оплата не прошла» с причиной, кнопками
  «Повторить» и «Выбрать другой способ». Деньги не списаны — снимаем панику.
- **Адрес вне зоны курьера** — предупреждение прямо в блоке доставки и кнопка-подсказка
  «Выбрать пункт выдачи». Не тупик, а маршрут к решению.

Для ревью в прототип встроена панель «Демо-сценарии» — переключает состояния
(авторизован / новый гость / ошибка оплаты / адрес вне зоны), чтобы проверяющий
мог потрогать каждый кейс без скрытых триггеров.

> Пустую корзину как edge-кейс исключили осознанно: на чекаут без товаров попасть
> нельзя, проектировать это состояние смысла нет.

---

## 8. Адаптивность

Прототип спроектирован mobile-first. На десктопе (≥900px) чекаут разворачивается
в две колонки: формы слева, липкий денежный виджет справа — стандарт веб-чекаута
(Ozon, Маркет). Это честное прочтение «mobile-first web»: одна верстка работает
и на телефоне, и на широком экране.

## 9. Гипотезы для проверки на реальном трафике

Каждую проверяю A/B-тестом, метрика успеха — рядом.

- **Smart CTA vs мёртвая блокировка** → рост completion rate на шаге ввода данных
  у новых пользователей (цель — снизить drop-off на шаге адреса/получателя).
- **Денежный виджет с промо рассрочки на виду** → рост доли выбора Сплита и среднего
  чека (прямой эффект на выручку).
- **Trust-блок у кнопки** → снижение отвала на шаге оплаты для заказов с высоким чеком.
- **Выбор ПВЗ на карте vs списком** → доля ошибочно выбранных пунктов (возвраты/переадресации).
- **СБП −3% бейджем в карточке метода** → сдвиг доли оплат в сторону СБП (ниже эквайринг).
